1.请说一说软件开发流程

项目启动 -> 需求分析 -> 需求概要 ->详细设计 -> 代码实现 ->集成测试 -> 系统部署 ->验收测试 -> 正式上线

2.解释 Java 中的 final、finally 和 finalize 的区别，并举例说明它们的使用场景。

final 是一个关键字，用于修饰类、方法或变量，表示不可变性。例如，修饰变量时使其值不可更改（如 final int MAX = 100;）；修饰方法时禁止子类重写（如 public final void process()）；修饰类时禁止继承（如 public final class Utility）。在项目中，我用 final 定义常量，如数据库连接池的最大连接数。

finally 是 try-catch 结构中的一部分，确保代码块无论是否抛出异常都会执行，常用于资源清理。例如，在数据库操作中，我会在 finally 中关闭 Connection 和 Statement 对象：

Connection conn = null;
try {
    conn = dataSource.getConnection();
    // 操作数据库
} catch (SQLException e) {
    log.error("数据库操作失败", e);
} finally {
    if (conn != null) conn.close();
}

finalize 是 Object 类的方法，在对象被垃圾回收前由 GC 调用，用于清理资源，但不推荐使用（Java 9 已标记为过时）。我从未在生产代码中使用 finalize，因为它执行时机不确定，优先使用 try-with-resources 确保资源管理。

3.描述 SOLID 原则中的“单一职责原则”和“开闭原则”，并结合实际项目中的例子说明如何应用。

**单一职责原则 (SRP)**：一个类只负责一项职责，降低代码耦合。例如，在一个订单管理系统中，我将订单创建逻辑放在 OrderService 类中，支付逻辑放在 PaymentService 类中，而不是将所有逻辑混杂在一个类中。这提高了代码可维护性，便于单独测试。

**开闭原则 (OCP)**：软件实体应对扩展开放，对修改关闭。在一个电商项目中，我设计了一个支付模块，使用策略模式支持多种支付方式（如微信、支付宝）。通过定义 PaymentStrategy 接口和实现类（如 WeChatPayment、AlipayPayment），新支付方式只需新增实现类，无需修改核心支付逻辑：

public interface PaymentStrategy {
    void pay(Order order);
}
public class WeChatPayment implements PaymentStrategy {
    public void pay(Order order) { /* 微信支付逻辑 */ }
}
public class PaymentService {
    private PaymentStrategy strategy;
    public void setStrategy(PaymentStrategy strategy) {
        this.strategy = strategy;
    }
    public void processPayment(Order order) {
        strategy.pay(order);
    }
}

4.在多线程编程中，synchronized 和 ReentrantLock 的区别是什么？在什么场景下你会选择使用 ReentrantLock？

区别：

  • synchronized 是 Java 关键字，自动管理锁的获取和释放，代码简洁，但功能有限。
  • ReentrantLock 是 java.util.concurrent.locks 包中的类，支持可重入锁、公平锁、条件变量（如 Condition）、中断响应等高级功能。
  • synchronized 无法中断等待锁的线程，而 ReentrantLock 提供 lockInterruptibly() 方法。
  • ReentrantLock 支持非阻塞锁获取（如 tryLock()），而 synchronized 不支持。

选择场景：

  • 使用 synchronized：简单同步场景，如保护共享资源（如计数器）。

  • 使用

    ReentrantLock

    ：需要高级功能时，例如：

    • 条件等待：在一个库存管理项目中，我使用 ReentrantLock 和 Condition 实现生产者-消费者模型，精确控制线程等待和唤醒。
    • 中断响应：在高并发任务处理中，允许线程被中断以避免死锁。
    • 公平锁：在任务调度场景中，设置 ReentrantLock 为公平锁以确保线程按顺序获取锁。
ReentrantLock lock = new ReentrantLock();
Condition condition = lock.newCondition();
lock.lock();
try {
    while (/* 条件不满足 */) condition.await();
    // 操作
    condition.signal();
} finally {
    lock.unlock();
}

5.描述 JVM 的内存模型，包括堆、栈、方法区等区域的作用，以及对象在这些区域中的生命周期。

  • JVM 内存模型：

    • **堆 (Heap)**：存储对象实例和数组，由所有线程共享。分为新生代（Eden、Survivor）和老年代，垃圾回收主要发生在此。
    • **栈 (Stack)**：每个线程私有，存储方法调用的栈帧（局部变量、方法参数、返回地址）。栈帧随方法调用创建，方法结束销毁。
    • **方法区 (Method Area)**：存储类信息、常量池、静态变量等，Java 8 后由元空间 (Metaspace) 实现，位于本地内存。
    • **程序计数器 (PC Register)**：线程私有，记录当前线程执行的字节码指令地址。
    • **本地方法栈 (Native Method Stack)**：支持本地方法（JNI）调用。

    对象生命周期：

    1. 对象在堆中分配（通常在 Eden 区）。
    2. 方法调用时，对象的引用存储在栈的局部变量表中。
    3. 方法结束后，栈帧弹出，引用失效，对象在堆中等待垃圾回收。
    4. 若对象无引用，GC 回收（如 Minor GC 或 Full GC）。

    项目经验：在优化一个高并发系统时，我通过调整 -Xms 和 -Xmx 设置堆大小，设置 -XX:NewRatio 优化新生代和老年代比例，减少 Full GC 频率。

6.如果一个 Java 应用出现内存泄漏，你会如何排查？请说明使用哪些工具以及具体步骤。

步骤：

  1. 监控现象：通过日志或监控工具（如 Prometheus）发现内存使用持续增长。

  2. 生成堆转储：使用 jmap -dump:live,format=b,file=heap.bin 或 VisualVM 生成堆转储文件。

  3. 分析堆转储

    ：使用 Eclipse MAT 或 JVisualVM 分析：

    • 查看“Leak Suspects”报告，定位占用内存最多的对象。
    • 检查引用链，找出未释放的对象（如缓存、静态集合）。

  4. 检查代码：定位问题代码（如未关闭的资源、未清理的 HashMap）。

  5. 验证修复：修复后通过压力测试验证内存使用是否稳定。

项目案例：在一个用户管理系统中，我发现内存持续增长。通过 MAT 分析，发现一个静态 HashMap 未清理旧用户数据。我添加了定时清理逻辑，问题解决。

工具：JVisualVM、MAT、jstat（监控 GC 频率）、jmap（生成堆转储）。

7.解释 Spring 的 IOC 和 DI 的实现原理，以及它们如何帮助解耦代码？举例说明一个实际项目中如何使用 @Autowired 或 @Resource。

IOC（控制反转）：Spring 通过 IOC 容器（ApplicationContext）管理对象的创建和生命周期，开发者无需手动实例化对象，降低耦合。

DI（依赖注入）：IOC 容器通过构造函数、Setter 或字段注入依赖对象。Spring 使用反射机制扫描 @Component、@Service 等注解，构建 Bean 定义，然后通过 @Autowired 或 @Resource 注入依赖。

解耦：通过配置或注解，类之间通过接口依赖，而不是直接依赖具体实现，便于替换和测试。

项目示例：在一个订单管理系统中，我定义了 OrderService 和 PaymentService：

@Service
public class OrderService {
    @Autowired
    private PaymentService paymentService;
    public void createOrder(Order order) {
        paymentService.process(order);
    }
}
@Service
public class PaymentService {
    public void process(Order order) { /* 支付逻辑 */ }
}

这里，OrderService 不需要手动实例化 PaymentService，Spring 自动注入，降低了耦合，方便替换支付实现。

8.Spring AOP 的核心概念是什么？请描述一个你使用 AOP 解决实际问题的场景。

核心概念：

  • Aspect：切面，封装横切关注点（如日志、事务）。
  • Join Point：程序执行的特定点（如方法调用）。
  • Advice：切面在连接点执行的动作（如 Before、After）。
  • Pointcut：定义哪些连接点需要应用 Advice。
  • AOP 代理：Spring 使用 JDK 动态代理或 CGLIB 生成代理对象。

项目案例：在一个 REST API 项目中，我使用 AOP 记录所有 Controller 方法的执行时间：

@Aspect
@Component
public class LoggingAspect {
    @Around("execution(* com.example.controller..*.*(..))")
    public Object logExecutionTime(ProceedingJoinPoint joinPoint) throws Throwable {
        long start = System.currentTimeMillis();
        Object result = joinPoint.proceed();
        long duration = System.currentTimeMillis() - start;
        log.info("方法 {} 执行耗时 {} ms", joinPoint.getSignature(), duration);
        return result;
    }
}

这通过 @Around 切面统一记录了日志，避免在每个方法中重复编写代码。

9.Spring Boot 的自动配置是如何实现的？如果需要自定义一个 Starter，应该如何设计？

自动配置原理：

  • Spring Boot 通过 @EnableAutoConfiguration 启用自动配置。
  • 扫描 META-INF/spring.factories 文件，加载 AutoConfiguration 类。
  • 使用 @Conditional 注解（如 @ConditionalOnClass、@ConditionalOnMissingBean）决定是否加载 Bean。

自定义 Starter：

  1. 创建模块（如 my-starter），添加依赖（如核心库）。
  2. 定义配置类：
@Configuration
@ConditionalOnClass(MyService.class)
public class MyAutoConfiguration {
    @Bean
    @ConditionalOnMissingBean
    public MyService myService() {
        return new MyService();
    }
}

3.在 resources/META-INF/spring.factories 中注册：

org.springframework.boot.autoconfigure.EnableAutoConfiguration=com.example.MyAutoConfiguration

4.打包发布为 Maven 依赖。

项目经验：我开发了一个日志 Starter，自动配置 Logback 和自定义日志格式，供多个项目复用。

10.你提到熟悉 React 和 Vue，请比较两者的核心理念，并说明在什么场景下选择哪一个框架。

React：

  • 核心理念：基于组件化、单向数据流和虚拟 DOM，使用 JSX 编写 UI，强调函数式编程。
  • 优点：灵活，生态丰富（如 Redux、React Router），适合复杂单页应用。
  • 场景：需要高度自定义、状态管理复杂的项目，如仪表盘系统。

Vue：

  • 核心理念：响应式数据绑定、模板语法、双向绑定（v-model），易上手。
  • 优点：学习曲线平缓，适合快速开发中小型项目。
  • 场景：快速原型开发或表单密集型应用，如内部管理系统。

项目选择：在一个微前端项目中，我选择 React 配合 Qiankun，因为其灵活性和社区支持适合模块化开发；在一个快速迭代的后台管理系统中，我使用 Vue 和 ElementUI 以提高开发效率。

11.Redis 如何实现分布式锁？请写一个简单的代码示例，并说明可能存在的风险。

实现原理：使用 Redis 的 SETNX（Set if Not Exists）设置锁，结合 EXPIRE 设置过期时间避免死锁。

代码示例：

public class DistributedLock {
    private Jedis jedis;
    private static final String LOCK_KEY = "lock:resource";
    private static final String LOCK_VALUE = UUID.randomUUID().toString();
    private static final int EXPIRE_SECONDS = 30;

    public boolean acquireLock() {
        // SETNX 尝试获取锁
        if (jedis.setnx(LOCK_KEY, LOCK_VALUE) == 1) {
            jedis.expire(LOCK_KEY, EXPIRE_SECONDS);
            return true;
        }
        return false;
    }

    public void releaseLock() {
        // 确保只有锁持有者能释放
        if (LOCK_VALUE.equals(jedis.get(LOCK_KEY))) {
            jedis.del(LOCK_KEY);
        }
    }
}

风险：

  • 锁过期：如果业务逻辑执行时间超过锁的过期时间，可能导致锁提前释放，需延长锁时间或使用 Redisson 的看门狗机制。
  • 非原子性：SETNX 和 EXPIRE 非原子，Redis 2.8+ 可用 SET key value NX PX timeout 解决。
  • 主从不一致：Redis 主从复制可能导致锁丢失，使用 Redlock 算法提高可靠性。

项目经验：在秒杀系统中，我使用 Redisson 实现分布式锁，确保库存扣减的线程安全。

12.请分享一个你参与的基于 Spring Boot 和微前端的项目，描述项目的架构设计、技术选型以及你负责的部分。

项目背景：一个企业级项目管理系统，支持项目创建、任务分配和进度跟踪。

架构设计：

  • 后端：Spring Boot + Spring Cloud Alibaba（Nacos 服务注册与配置、Gateway 网关）、MySQL + Redis。
  • 前端：React + Qiankun 微前端，Material-UI 组件库。
  • 部署：Docker + Kubernetes，Jenkins 实现 CI/CD。

技术选型：

  • Spring Boot 提供快速开发，Nacos 管理微服务配置。
  • Qiankun 实现微前端模块化，React 提供灵活的组件化开发。
  • Redis 缓存热点数据，MySQL 存储项目信息。

我的职责：

  • 开发了项目创建微服务，包含表单验证和 REST API 设计。
  • 实现微前端模块（项目仪表盘、项目详情），通过 Qiankun 集成到宿主应用。
  • 配置 Jenkins 流水线，从代码提交到 Docker 镜像构建和部署。
  • 优化 MySQL 查询性能，使用索引和分表降低响应时间。

成果：系统支持 1000+ 用户并发访问，平均响应时间 < 200ms。

14.在 Spring Cloud 微服务项目中，如何处理服务间通信？会选择 Feign 还是 RabbitMQ，为什么？

服务间通信方式：

  • 同步通信：使用 Feign 或 RestTemplate 调用 REST API，适合实时性要求高的场景（如查询用户信息）。
  • 异步通信：使用 RabbitMQ 或 Kafka 处理消息队列，适合解耦和异步处理（如订单状态更新）。

选择依据：

  • Feign：声明式 HTTP 客户端，集成 Spring Cloud OpenFeign，适合微服务间快速调用。我在一个订单系统中使用 Feign 调用支付服务，配合 Nacos 实现服务发现，代码简洁且易于维护：
@FeignClient(name = "payment-service")
public interface PaymentClient {
    @PostMapping("/pay")
    String processPayment(@RequestBody Order order);
}

RabbitMQ：适合高吞吐、异步场景。在一个日志系统中，我使用 RabbitMQ 收集服务日志，消费者异步写入 Elasticsearch，避免阻塞主线程。

项目选择：在实时性要求高的场景（如支付验证），选择 Feign；在解耦和批量处理场景（如日志收集），选择 RabbitMQ。

15.描述你使用 Jenkins 构建 CI/CD 流水线的经验，包括代码提交到上线部署的完整流程。

项目背景：一个 Spring Boot 微服务项目，部署在阿里云 ECS。

CI/CD 流程：

  1. 代码提交：开发在 Git 提交代码到 GitLab 分支（如 dev）。

  2. 触发构建：Jenkins 配置 Webhook，监听 GitLab 推送事件，触发流水线。

  3. 构建阶段

    ：

    • 使用 Maven 执行 mvn clean package，生成 JAR 文件。
    • 运行单元测试（JUnit + Mockito），确保代码质量。

  4. 镜像构建：Jenkins 调用 Dockerfile，构建 Docker 镜像并推送至阿里云镜像仓库：

    FROM openjdk:11
COPY target/app.jar /app.jar
ENTRYPOINT ["java", "-jar", "/app.jar"]

  5. 部署阶段：通过 SSH 登录 ECS，执行 docker-compose.yml 更新服务：

    version: '3'
services:
  app:
    image: registry.cn-hangzhou.aliyuncs.com/myrepo/app:latest
    ports:
      - "8080:8080"

  6. 验证：Jenkins 调用健康检查接口（/actuator/health），确认服务正常。

成果：实现代码提交到部署的自动化，部署时间从 30 分钟缩短到 5 分钟。
