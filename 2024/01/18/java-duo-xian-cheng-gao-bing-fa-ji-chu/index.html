1.高并发相关概念

1.1进程与线程相关概念

  1. 独立性：
    • 进程：进程是操作系统分配资源（如内存、文件描述符等）的基本单位，每个进程都有自己独立的地址空间和系统资源。进程之间相互隔离，一个进程崩溃不会直接影响到其他进程。
    • 线程：线程是进程内的一个执行流，它共享进程的地址空间和资源。一个进程可以包含多个线程，这些线程可以并行执行，共享数据和代码。
  2. 资源开销：
    • 进程：创建、维护和销毁进程的开销通常比线程大，因为操作系统需要为每个进程分配独立的资源。
    • 线程：创建和销毁线程的开销较小，因为线程共享进程的资源，所以线程间的切换也比进程间的切换更快。
  3. 通信：
    • 进程：进程间通信（IPC）相对复杂，可能需要使用管道、消息队列、信号量、共享内存等机制。
    • 线程：线程间通信更简单，因为它们共享内存空间，可以直接访问彼此的数据。
  4. 安全性：
    • 进程：由于进程间相互隔离，一个进程的错误或安全漏洞通常不会影响其他进程。
    • 线程：由于线程共享内存，一个线程的错误或安全漏洞可能会影响同一进程中的其他线程。
  5. 并发性：
    • 进程：多进程可以实现并发执行，但进程间的并发需要操作系统进行调度。
    • 线程：多线程在同一进程内实现并发执行，线程间的并发执行通常由操作系统的线程调度器管理。
  6. 生命周期：
    • 进程：进程的生命周期通常包括创建、执行、等待、终止等状态。
    • 线程：线程的生命周期与进程类似，但由于线程是进程的一部分，其生命周期也受到进程状态的影响。

关系：一个进程里面可能包含多个线程。

1.2主线程和子线程

  1. 主线程：
    • 定义：在大多数操作系统中，程序启动时自动创建的线程被称为主线程。它是程序的入口点，通常负责初始化程序、调用主要的逻辑和处理用户界面等。
    • 作用：主线程是程序执行的起点，它负责启动程序并执行程序的主要逻辑。在某些情况下，主线程也负责管理其他线程的创建和销毁。
  2. 子线程：
    • 定义：子线程是由主线程或其他线程创建的线程。它们通常用于执行特定的任务，如后台处理、异步操作或并行计算。
    • 作用：子线程可以并行执行，从而提高程序的性能。它们可以执行耗时的任务，如文件读写、网络请求或复杂的计算，而不会阻塞主线程，使得用户界面保持响应。

举例：在JVM启动时会创建一个主线程，这个主线程执行main方法，那么在main方法中创建的线程Thread就是在主线程中创建的子线程.

关系：如果B是A的子线程(但B不是A的守护线程)，那么A和B两个线程的周期互不影响，也就是A线程的结束不会造成B线程的结束，反之亦然.

1.3串行，并发，并行

  1. 串行（Serial）：
    • 串行指的是任务按照顺序一个接一个地执行。在串行执行中，前一个任务必须完成后，下一个任务才能开始。这种方式简单直接，但效率较低，因为它不利用多核处理器或多处理器的优势。
    • 在计算机系统中，串行执行通常意味着只有一个CPU核心或进程在工作，而其他核心或进程处于空闲状态。
  2. 并发（Concurrency）：
    • 并发是指多个任务在同一时间段内启动，但不一定同时执行。在并发系统中，任务可以重叠执行，这意味着它们可以共享时间片，但不一定同时在执行。
    • 并发通常涉及到任务调度，操作系统会根据优先级、资源可用性等因素来决定哪个任务应该在何时执行。并发可以提高系统的响应性和吞吐量，但也需要更复杂的同步和调度机制来避免资源竞争。
  3. 并行（Parallelism）：
    • 并行是指多个任务在同一时刻同时执行。在并行系统中，任务被分配到多个处理器或核心上，每个处理器或核心同时执行不同的任务部分。
    • 并行执行可以显著提高计算密集型任务的执行速度，因为它充分利用了硬件的并行处理能力。然而，实现并行执行需要任务能够分解成可以独立执行的部分，并且可能需要复杂的同步和通信机制来协调不同部分的执行。

2.线程的创建和启动

2.1线程的创建

2.1.1实现Runnable接口

public class MyRunnable implements Runnable {
    public void run() {
        // 线程执行的代码
    }
}

2.1.2继承Thread类并重写run()方法

public class MyThread extends Thread {
    public void run() {
        // 线程执行的代码
    }
}

2.2线程的启动

  • 对于继承Thread类的对象，调用start()方法启动线程：

    MyThread myThread = new MyThread();
myThread.start();

  • 对于实现Runnable接口的对象，需要先创建Thread对象，然后调用start()方法：

    MyRunnable myRunnable = new MyRunnable();
Thread thread = new Thread(myRunnable);
thread.start();

调用start()方法启动线程，实质就是请求JVM运行相应的线程，这个线程具体什么时候运行，由线程调度器决定

2.3start()和run()的区别

1.start()方法的作用：开启新的线程，并且自动执行线程中的run()方法，且run方法只执行一次
2.run()方法：相当于普通的方法，可以直接调用，但是不会创建新的线程,谁的线程调用run()就在谁的线程中运行run()

3.线程的常用方法

3.1 currentThread()方法获取当前线程
3.2 setName()/getName()方法设置和获取线程名称
3.3 isAlive()判断线程是否存活
3.4 yield()让正在运行的线程让出CPU资源
3.5 setPriority()设置线程的优先级
设置线程优先级一定要谨慎，避免造成饿死现象，所谓饿死现象就是，一些线程一直得不到CPU资源，从而一直处于等待状态

3.5 interrupt()中断线程
该方法仅仅只是给线程设置了一个中断标志告诉JVM此线程可以中断了，但并不意味这线程就立刻中断了，可能仍然在执行

3.6 setDaemon()设置守护线程
守护线程特点：
1.守护线程不能单独执行，必须依赖于普通非守护线程
2.当非守护线程结束时，守护线程也结束了，也即守护线程的生命周期依赖于它所守护的非守护线程

4.线程的生命周期

线程的生命周期包含5个阶段，包括：新建、就绪、运行、阻塞、销毁。

  • 新建：就是刚使用new方法，new出来的线程；
  • 就绪：就是调用的线程的start()方法后，这时候线程处于等待CPU分配资源阶段，谁先抢的CPU资源，谁开始执行;
  • 运行：当就绪的线程被调度并获得CPU资源时，便进入运行状态，run方法定义了线程的操作和功能;
  • 阻塞：在运行状态的时候，可能因为某些原因导致运行状态的线程变成了阻塞状态，比如sleep()、wait()之后线程就处于了阻塞状态，这个时候需要其他机制将处于阻塞状态的线程唤醒，比如调用notify或者notifyAll()方法。唤醒的线程不会立刻执行run方法，它们要再次等待CPU分配资源进入运行状态;
  • 销毁：如果线程正常执行完毕后或线程被提前强制性的终止或出现异常导致结束，那么线程就要被销毁，释放资源;

5.线程的优势和缺点

优势：

  1. 资源共享：同一进程内的多个线程可以共享进程的资源，如内存空间、文件句柄等，这使得线程间的通信和数据交换更加高效。
  2. 并发执行：多线程可以在同一进程内并发执行，提高了程序的执行效率和响应速度。
  3. 开销较小：创建线程的开销通常比创建进程要小，因为线程共享进程的资源，不需要额外的系统资源分配。
  4. 提高效率：对于I/O密集型任务，多线程可以避免单个线程等待I/O操作完成，从而提高整体的执行效率。
  5. 简化编程模型：多线程编程可以简化某些并发问题的编程模型，使得程序员更容易处理并发任务。

缺点：

  1. 同步问题：多线程编程中，线程之间的同步是一大挑战。如果处理不当，可能会导致竞态条件（race conditions）、死锁（deadlocks）等问题。
  2. 上下文切换：虽然线程的开销小，但频繁的线程切换（context switching）会消耗CPU资源，降低系统性能。
  3. 复杂性增加：多线程编程的复杂性高于单线程编程，需要更多的编程技巧和对并发控制的理解。
  4. 调试难度：多线程程序的调试通常比单线程程序困难，因为线程的执行顺序和状态可能难以预测。
  5. 资源限制：虽然线程共享进程资源，但如果线程数量过多，可能会导致单个进程的资源耗尽，影响系统稳定性。

6.线程安全问题

6.1竞态条件（Race Condition)

当两个或多个线程尝试访问共享资源，而这些操作的执行顺序会影响程序结果时，就可能发生竞态条件。例如，两个线程同时读取一个共享变量，然后各自进行计算，最后写回该变量，可能导致最终结果不是预期的。竞态条件的一个典型例子是银行账户的并发存取操作。假设有两个线程，线程A和线程B，它们都试图从同一个账户中取款。如果它们同时读取账户余额，然后各自减去自己的取款金额，最后再更新账户余额，那么最终的余额可能不会是它们期望的总和，因为它们可能基于相同的初始余额进行计算。

下面是一个简化的Java代码示例，展示了竞态条件：

public class Account {
    private int balance = 1000;

    public void withdraw(int amount) {
        // 假设这里没有同步机制
        balance -= amount;
    }
}

public class Main {
    public static void main(String[] args) {
        Account account = new Account();
        Thread t1 = new Thread(() -> account.withdraw(500));
        Thread t2 = new Thread(() -> account.withdraw(200));

        t1.start();
        t2.start();
    }
}

在这个例子中，如果t1和t2同时执行withdraw方法，它们可能会读取相同的初始余额（1000），然后分别减去500和200，导致最终余额变为300，而不是预期的800。

在上述Account类的withdraw方法中，可以通过添加synchronized关键字来解决竞态条件：

public synchronized void withdraw(int amount) {
    balance -= amount;
}

或者使用AtomicInteger：

复制import java.util.concurrent.atomic.AtomicInteger;

public class Account {
    private AtomicInteger balance = new AtomicInteger(1000);

    public void withdraw(int amount) {
        balance.addAndGet(-amount);
    }
}

这些方法可以确保在任何给定时间，只有一个线程可以修改账户余额，从而避免了竞态条件。

6.2死锁（Deadlock）

当两个或多个线程在执行过程中，因争夺资源而造成的一种互相等待的现象，若无外力作用，它们都将无法继续执行。死锁通常发生在线程同时请求多个资源时，且这些资源的请求和释放顺序不一致。

死锁的四个必要条件是：

  1. 互斥条件（Mutual Exclusion）：至少有一个资源（如锁）在同一时间只能被一个线程使用。
  2. 占有并等待（Hold and Wait）：一个线程持有至少一个资源，同时等待其他线程释放它所占有的资源。
  3. 不可抢占（No Preemption）：线程占有的资源在未使用完之前，不能被其他线程抢占。
  4. 循环等待（Circular Wait）：存在一组线程，其中每个线程都在等待下一个线程释放资源。

下面是一个简单的Java死锁示例：

public class DeadlockExample {
    private static final Object lock1 = new Object();
    private static final Object lock2 = new Object();

    public static void main(String[] args) {
        Thread thread1 = new Thread(() -> {
            synchronized (lock1) {
                System.out.println("Thread 1: locked lock1");
                try { Thread.sleep(100); } catch (InterruptedException e) {}
                synchronized (lock2) {
                    System.out.println("Thread 1: locked lock2");
                }
            }
        });

        Thread thread2 = new Thread(() -> {
            synchronized (lock2) {
                System.out.println("Thread 2: locked lock2");
                try { Thread.sleep(100); } catch (InterruptedException e) {}
                synchronized (lock1) {
                    System.out.println("Thread 2: locked lock1");
                }
            }
        });

        thread1.start();
        thread2.start();
    }
}

在这个例子中，thread1 和 thread2 分别尝试获取两个锁 lock1 和 lock2。由于它们试图以不同的顺序获取这两个锁，它们最终会互相等待对方释放锁，导致死锁。

为了避免死锁，可以采取以下策略：

  1. 资源分配图：使用资源分配图来检测循环等待条件。
  2. 资源有序分配：为所有资源分配一个全局唯一的顺序，并按照这个顺序请求资源。
  3. 死锁预防：破坏死锁的四个必要条件之一。例如，可以设计资源为可抢占的，或者确保线程不会同时占有多个资源。
  4. 死锁检测与恢复：在系统中实现死锁检测机制，并在检测到死锁时进行恢复，例如通过终止或回滚某些线程。
  5. 使用锁的尝试获取（tryLock）：在尝试获取锁时，如果锁已被其他线程持有，可以选择等待或放弃，而不是立即进入阻塞状态。

为了解决死锁问题，我们可以通过使用Java并发包中的ReentrantLock来避免死锁。ReentrantLock提供了一个tryLock()方法，这个方法在尝试获取锁时不会阻塞线程，而是返回一个布尔值，指示是否成功获取了锁。如果锁已被其他线程持有，tryLock()会立即返回false，这样线程就可以选择重试或者执行其他操作。

下面是一个使用ReentrantLock来避免死锁的示例：

import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

public class DeadlockAvoidanceExample {
    private static final Lock lock1 = new ReentrantLock();
    private static final Lock lock2 = new ReentrantLock();

    public static void main(String[] args) {
        Thread thread1 = new Thread(() -> {
            try {
                if (lock1.tryLock()) {
                    try {
                        System.out.println("Thread 1: locked lock1");
                        // 模拟耗时操作
                        Thread.sleep(100);
                        if (lock2.tryLock()) {
                            try {
                                System.out.println("Thread 1: locked lock2");
                                // 执行需要两个锁的代码
                            } finally {
                                lock2.unlock();
                            }
                        }
                    } finally {
                        lock1.unlock();
                    }
                } else {
                    System.out.println("Thread 1: failed to lock lock1");
                }
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        });

        Thread thread2 = new Thread(() -> {
            try {
                if (lock2.tryLock()) {
                    try {
                        System.out.println("Thread 2: locked lock2");
                        // 模拟耗时操作
                        Thread.sleep(100);
                        if (lock1.tryLock()) {
                            try {
                                System.out.println("Thread 2: locked lock1");
                                // 执行需要两个锁的代码
                            } finally {
                                lock1.unlock();
                            }
                        }
                    } finally {
                        lock2.unlock();
                    }
                } else {
                    System.out.println("Thread 2: failed to lock lock2");
                }
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        });

        thread1.start();
        thread2.start();
    }
}

在这个示例中，两个线程尝试获取两个锁。如果一个线程无法获取第一个锁，它会立即释放所有已获取的锁（如果有的话），并重试。这样可以避免线程因为等待其他线程释放锁而陷入死锁。

请注意，这个策略并不保证完全避免死锁，但它确实减少了死锁的可能性。在实际应用中，你可能需要根据具体情况调整策略，例如设置重试次数、引入超时机制等。

6.3活锁（Livelock）

与死锁类似，活锁是指两个或多个线程在尝试执行任务时，由于条件不断变化，导致它们不断地尝试进入执行状态但又不断被中断，从而无法完成任何有用的工作。

活锁的一个典型例子是两个线程同时尝试进入一个临界区，但它们总是检测到对方的存在并主动让步，结果两者都无法进入临界区。这与死锁不同，死锁是线程因为互相等待对方释放资源而停滞不前，而活锁中的线程则是在不断地尝试和失败。

下面是一个简单的活锁示例：

public class LivelockExample {
    private static boolean flag = false;

    public static void main(String[] args) {
        Thread thread1 = new Thread(() -> {
            while (true) {
                if (!flag) {
                    flag = true;
                    // 尝试执行任务
                    // ...
                    flag = false;
                    break; // 假设任务执行成功，退出循环
                }
            }
        });

        Thread thread2 = new Thread(() -> {
            while (true) {
                if (!flag) {
                    flag = true;
                    // 尝试执行任务
                    // ...
                    flag = false;
                    break; // 假设任务执行成功，退出循环
                }
            }
        });

        thread1.start();
        thread2.start();
    }
}

在这个例子中，两个线程都在不断地检查flag变量，如果它为false，线程会尝试设置它为true并执行任务。然而，由于两个线程几乎同时检查和设置flag，它们会不断地相互覆盖对方的操作，导致任务永远无法执行。

解决活锁的方法通常包括：

  1. 引入随机化：在尝试操作之间引入随机的延迟，这样线程在尝试失败后会等待一段时间再重试，减少同时尝试的概率。
  2. 设置超时：为尝试操作设置超时时间，如果超过这个时间还没有成功，线程可以选择放弃当前操作或采取其他策略。
  3. 使用更复杂的同步机制：例如，使用计数器来跟踪尝试次数，当达到一定次数后，线程可以选择让步或采取其他策略。
  4. 设计更好的并发策略：重新设计并发逻辑，确保线程在尝试操作时能够更有效地协调，避免活锁的发生。

活锁的解决通常需要对并发逻辑进行细致的分析和调整，以确保线程能够有效地完成任务。

6.4饥饿（Starvation）

当一个线程因为其他线程的优先级更高或者资源分配策略不当，长时间无法获得执行机会，就可能导致饥饿。这可能导致线程永远无法执行其任务。

线程饥饿的一个典型示例是在一个多线程环境中，有一个或多个线程因为其他线程频繁地获取资源而无法执行。以下是一个简单的Java示例，展示了线程饥饿的情况：

复制import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

public class ThreadStarvationExample {
    private static final Lock lock = new ReentrantLock();

    public static void main(String[] args) {
        // 创建两个线程，一个线程（线程A）频繁地获取锁，另一个线程（线程B）尝试获取锁
        Thread threadA = new Thread(() -> {
            while (true) {
                // 线程A频繁地获取锁
                lock.lock();
                try {
                    System.out.println("Thread A: Acquired lock, doing work...");
                    // 模拟工作
                    Thread.sleep(100);
                } finally {
                    lock.unlock();
                }
            }
        });

        Thread threadB = new Thread(() -> {
            while (true) {
                // 线程B尝试获取锁，但由于线程A频繁地持有锁，线程B很难获取到
                try {
                    if (lock.tryLock()) {
                        System.out.println("Thread B: Acquired lock, doing work...");
                        // 模拟工作
                        Thread.sleep(100);
                        lock.unlock();
                    }
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            }
        });

        // 启动线程
        threadA.start();
        threadB.start();
    }
}

在这个例子中，线程A和线程B都尝试获取同一个锁。线程A通过lock.lock()方法获取锁，并且由于它在锁的保护下执行了一个短暂的睡眠，这使得线程B很难有机会获取到锁。即使线程B使用tryLock()方法尝试获取锁，但由于线程A几乎总是持有锁，线程B几乎不会有机会执行其工作。

为了解决这个问题，可以采取以下措施：

  1. 限制锁的持有时间：确保线程在锁的保护下执行的操作尽可能短，以便其他线程有机会获取锁。
  2. 使用公平锁：如前所述，使用ReentrantLock(true)可以创建一个公平锁，这样所有线程按照请求锁的顺序获取锁。
  3. 资源池：如果锁是为了保护资源池中的资源，可以考虑使用资源池来管理资源，这样每个线程都可以从池中获取资源，而不是竞争同一个锁。
  4. 优先级调整：在某些情况下，可以通过调整线程的优先级来解决饥饿问题，但这可能会导致其他问题，如优先级反转。

6.5优先级反转（Priority Inversion）

线程优先级反转（Priority Inversion）是多线程编程中的一个问题，它发生在一个高优先级的线程因为等待一个低优先级的线程释放资源而阻塞时。如果这个低优先级的线程又被另一个更低优先级的线程阻塞，就形成了优先级反转。这种情况下，高优先级的线程无法执行，因为它在等待低优先级的线程，而低优先级的线程又在等待更低优先级的线程，这可能导致系统性能下降，甚至出现死锁。

优先级反转通常发生在使用优先级调度的系统中，尤其是在那些允许线程动态改变优先级的系统中。在Java中，虽然线程优先级不是调度的主要因素（Java主要依赖于CPU时间片和线程状态），但优先级反转的问题仍然可能发生。

以下是一个简化的优先级反转示例：

public class PriorityInversionExample {
    private static final Object lock1 = new Object();
    private static final Object lock2 = new Object();

    public static void main(String[] args) {
        // 创建三个线程，线程A是高优先级，线程B是中等优先级，线程C是低优先级
        Thread threadA = new Thread(() -> {
            synchronized (lock1) {
                System.out.println("Thread A: Acquired lock1, waiting for lock2...");
                synchronized (lock2) {
                    System.out.println("Thread A: Acquired lock2, doing high-priority work...");
                }
            }
        });

        Thread threadB = new Thread(() -> {
            synchronized (lock2) {
                System.out.println("Thread B: Acquired lock2, doing medium-priority work...");
                // 模拟工作
                try {
                    Thread.sleep(2000); // 假设线程B执行了较长时间的工作
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            }
        });

        Thread threadC = new Thread(() -> {
            // 线程C尝试获取lock2，但由于线程B已经持有，线程C会阻塞
            synchronized (lock2) {
                System.out.println("Thread C: Acquired lock2, doing low-priority work...");
                // 模拟工作
                try {
                    Thread.sleep(1000); // 假设线程C执行了较短时间的工作
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            }
        });

        // 启动线程
        threadA.setPriority(Thread.MAX_PRIORITY);
        threadB.setPriority(Thread.NORM_PRIORITY);
        threadC.setPriority(Thread.MIN_PRIORITY);

        threadA.start();
        threadB.start();
        threadC.start();
    }
}

在这个例子中，线程A（高优先级）需要获取两个锁（lock1和lock2）来执行高优先级的工作。线程B（中等优先级）持有lock2，而线程C（低优先级）尝试获取lock2。由于线程B持有lock2，线程A无法继续执行。同时，线程C尝试获取lock2，但由于线程B已经持有，线程C也会阻塞。这样，线程A（高优先级）就被线程C（低优先级）间接阻塞了，形成了优先级反转。

为了解决优先级反转问题，可以采取以下措施：

  1. 优先级继承：当一个低优先级的线程持有一个高优先级的线程所需的资源时，低优先级的线程可以临时继承高优先级线程的优先级，直到它释放资源。
  2. 优先级天花板：为共享资源设置一个优先级天花板，任何线程在获取这些资源时，其优先级都会被提升到这个天花板优先级。
  3. 避免使用优先级：在设计并发程序时，尽量避免使用线程优先级，或者确保所有线程都使用相同的优先级。
  4. 使用锁的公平模式：在Java中，使用ReentrantLock的公平模式（ReentrantLock(true)）可以减少优先级反转的风险，因为锁的获取是按照请求顺序进行的。
  5. 使用锁的尝试获取方法：使用tryLock()方法而不是lock()方法，这样在获取锁失败时，线程可以选择其他操作而不是阻塞。

在实际应用中，解决优先级反转问题通常需要对并发控制策略进行细致的设计和调整。

6.6不可重入（Non-reentrant）

线程不可重入（Non-reentrant）通常指的是一个线程在持有某个锁的情况下尝试再次获取同一个锁，而这种行为在某些同步机制下是不允许的。在Java中，标准的synchronized关键字和ReentrantLock类都是可重入锁（Reentrant Lock），这意味着同一个线程可以多次获取同一个锁而不会导致死锁。

不可重入锁在某些编程语言或同步机制中可能会出现，它们要求线程在持有锁时不能再次获取同一个锁。如果尝试这样做，可能会抛出异常或者导致死锁。

在Java中，如果你尝试使用synchronized块或方法在已经持有相同锁的线程中再次获取锁，这通常是安全的，因为Java的synchronized关键字支持可重入。但是，如果你在自定义的同步逻辑中不小心违反了可重入的原则，就可能导致问题。

下面是一个Java中可重入锁的示例：

public class ReentrantLockExample {
    private final Object lock = new Object();

    public void methodA() {
        synchronized (lock) {
            System.out.println("Entering method A");
            // 这里可以安全地调用其他需要同一个锁的方法
            methodB();
        }
    }

    public void methodB() {
        synchronized (lock) {
            System.out.println("Entering method B");
            // 这里也可以安全地调用其他需要同一个锁的方法
        }
    }
}

在这个例子中，methodA 和 methodB 都使用了同一个锁对象 lock。即使 methodA 已经持有了这个锁，它仍然可以安全地调用 methodB，因为 synchronized 关键字支持可重入。

如果你在自定义的同步逻辑中不小心违反了可重入原则，可能会导致问题。例如，如果你在自定义的锁类中没有正确处理重入情况，那么当线程尝试再次获取已经持有的锁时，可能会抛出异常或者导致死锁。

为了避免这种情况，你应该始终确保你的同步逻辑是可重入的，或者使用Java提供的可重入锁类，如ReentrantLock。这样，你就可以安全地在同一个线程中多次获取同一个锁，而不会破坏线程安全。

为了解决这些线程安全问题，开发者需要采取适当的同步机制，如互斥锁（mutexes）、信号量（semaphores）、条件变量（condition variables）、读写锁（read-write locks）、原子操作（atomic operations）等。此外，还可以使用线程安全的编程模式，如生产者-消费者模式、读者-写者模式等，以及使用线程安全的库和框架来简化并发编程的复杂性。线程安全问题主要表现为3个方面：原子性，可见性，有序性

7.线程间的通信

线程间的通信（Inter-Thread Communication）是指在多线程程序中，不同线程之间如何交换信息或者协调工作。在Java中，有多种机制可以实现线程间的通信，以下是一些常见的方法：

  1. 共享变量： 线程可以通过共享变量进行通信。这通常需要同步机制（如synchronized关键字或ReentrantLock）来确保数据的一致性和线程安全。

    public class SharedVariableExample {
    private static int sharedCounter = 0;

    public static void main(String[] args) {
        Thread thread1 = new Thread(() -> {
            for (int i = 0; i < 100; i++) {
                synchronized (SharedVariableExample.class) {
                    sharedCounter++;
                }
            }
        });

        Thread thread2 = new Thread(() -> {
            for (int i = 0; i < 100; i++) {
                synchronized (SharedVariableExample.class) {
                    sharedCounter++;
                }
            }
        });

        thread1.start();
        thread2.start();

        try {
            thread1.join();
            thread2.join();
        } catch (InterruptedException e) {
            e.printStackTrace();
        }

        System.out.println("Final counter value: " + sharedCounter);
    }
}

  2. wait() 和 notify()/notifyAll() 方法： 这些方法是Object类的一部分，可以用来实现线程间的等待和通知机制。一个线程可以通过调用wait()方法进入等待状态，直到另一个线程调用notify()或notifyAll()唤醒它。

    public class WaitNotifyExample {
    //volatile参数修饰变量保证其可见性
    public static volatile int number = 0;//蛋糕的数量
    public static final Object LOCK = new Object();//定义一个常量对象作为锁
    public static void main(String[] args) {

        for (int i = 1; i <= 10; i++) {
            //创建一个生产者线程
            new Thread(() -> {
                synchronized (LOCK) {
                    //当蛋糕不为0时，生产者进入等待
                    while (number != 0) {
                        try {
                            LOCK.wait();//生产者线程进入等待状态，释放锁资源
                        } catch (InterruptedException e) {
                            e.printStackTrace();
                        }
                    }
                    //当生产者被唤醒时，生产蛋糕
                    number++;
                    System.out.println(Thread.currentThread().getName() + "生产了一个蛋糕,蛋糕数量为:" + number);
                    //唤醒消费者线程消费
                    LOCK.notify();
                }
            }, "Cake Producer").start();

            //创建一个消费者线程
            new Thread(() -> {
                synchronized (LOCK) {
                    //当蛋糕为0时，消费者进入等待
                    while (number == 0) {
                        try {
                            LOCK.wait();//消费者线程进入等待状态，释放锁资源
                        } catch (InterruptedException e) {
                            e.printStackTrace();
                        }
                    }
                    //当消费者被唤醒时，消费蛋糕
                    number--;
                    System.out.println(Thread.currentThread().getName() + "消费了一个蛋糕,蛋糕数量为:" + number);
                    //唤醒生产者线程生产
                    LOCK.notify();
                }
            }, "Cake Consumer").start();
        }
    }
}

  3. CountDownLatch： CountDownLatch是一个同步辅助类，允许一个或多个线程等待一组操作在其他线程中完成。它提供了一个计数器，当计数器达到零时，等待的线程可以继续执行。

    public class CountDownLatchExample {
    public static void main(String[] args) {
        int threads = 5;
        CountDownLatch latch = new CountDownLatch(threads);

        ExecutorService executor = Executors.newFixedThreadPool(threads);
        for (int i = 0; i < threads; i++) {
            executor.submit(() -> {
                // 执行任务
                // ...
                latch.countDown(); // 任务完成后减少计数器
            });
        }

        try {
            latch.await(); // 主线程等待所有任务完成
        } catch (InterruptedException e) {
            e.printStackTrace();
        }

        System.out.println("All tasks completed.");
        executor.shutdown();
    }
}

  4. CyclicBarrier： CyclicBarrier允许一组线程相互等待，直到所有线程都到达某个屏障点。一旦所有线程都到达，它们就可以继续执行。

    public class CyclicBarrierExample {
    public static void main(String[] args) {
        CyclicBarrier barrier = new CyclicBarrier(5);

        ExecutorService executor = Executors.newFixedThreadPool(5);
        for (int i = 0; i < 5; i++) {
            executor.submit(() -> {
                // 执行任务
                // ...
                try {
                    barrier.await(); // 等待其他线程
                } catch (Exception e) {
                    e.printStackTrace();
                }
                // 继续执行
            });
        }

        executor.shutdown();
    }
}

  5. Exchanger： Exchanger允许两个线程交换数据。每个线程必须同时调用exchange()方法，否则会一直阻塞。

    public class ExchangerExample {
    public static void main(String[] args) {
        Exchanger<String> exchanger = new Exchanger<>();

        Thread thread1 = new Thread(() -> {
            String data1 = "Hello";
            try {
                String data2 = exchanger.exchange(data1);
                System.out.println("Thread 1 received: " + data2);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        });

        Thread thread2 = new Thread(() -> {
            String data2 = "World";
            try {
                String data1 = exchanger.exchange(data2);
                System.out.println("Thread 2 received: " + data1);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        });

        thread1.start();
        thread2.start();
    }
}

在实际应用中，选择合适的线程间通信机制取决于你的具体需求。例如，如果你需要线程等待其他线程完成特定的操作，CountDownLatch可能是一个好的选择。如果你需要线程在执行过程中同步，那么synchronized或ReentrantLock可能更适合。而对于复杂的同步场景，CyclicBarrier和Exchanger提供了更高级的同步控制。
