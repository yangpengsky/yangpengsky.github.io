springboot事务相关使用

前言

Spring Boot 是一个基于 Spring 框架的快速开发工具，它简化了 Spring 应用程序的创建和部署过程。在 Spring Boot 中，事务管理是一个重要的概念，它允许你控制数据访问操作（如数据库操作）的原子性、一致性、隔离性和持久性（ACID）。

事务管理方式

  1. 声明式事务管理（Declarative Transaction Management）： 这是 Spring 框架中最常用的事务管理方式，它通过注解来声明事务的属性，如传播行为、隔离级别等。这种方式不需要在业务逻辑中显式地处理事务的开始、提交和回滚，Spring 容器会自动处理这些操作。
    • 注解方式：使用 @Transactional 注解标记需要管理事务的方法。
    • XML 配置：在 Spring 的配置文件中声明事务管理器和事务属性。
  2. 编程式事务管理（Programmatic Transaction Management）： 这种方式需要开发者在代码中显式地管理事务的生命周期。这种方式提供了更多的灵活性，但同时也增加了代码的复杂性。
    • **使用 TransactionTemplate**：Spring 提供了 TransactionTemplate 类，它提供了一个编程式的 API 来管理事务。
    • **使用 PlatformTransactionManager**：直接使用 PlatformTransactionManager 接口来处理事务的开始、提交和回滚。

事务提交方式

  1. 自动提交（Auto Commit）： 在某些数据库管理系统中，如 MySQL，在默认情况下，每个 SQL 语句都是自动提交的。这意味着一旦执行了一个 SQL 语句，它就会立即被提交到数据库。如果需要控制事务的提交，可以显式地关闭自动提交功能，然后手动管理事务的提交和回滚。
  2. 显式提交（Explicit Commit）： 在事务处理中，开发者可以通过显式调用 COMMIT 语句来提交事务。这种方式要求开发者在事务的适当时刻手动发出提交指令。
  3. 隐式提交（Implicit Commit）： 在某些情况下，数据库系统可能会在特定操作后自动提交事务，例如在某些数据库中，当执行了 SELECT 语句后，如果紧接着执行了 INSERT、UPDATE 或 DELETE 操作，事务可能会被隐式提交。
  4. 事务结束时提交（Commit on Transaction End）： 在某些编程框架或 ORM（对象关系映射）工具中，事务的提交可能会在事务结束时自动处理。例如，在 Spring Boot 中，如果使用了声明式事务管理，事务的提交和回滚通常是由框架自动管理的，除非开发者通过编程式事务管理显式地控制。
  5. 事务超时提交（Commit on Timeout）： 在某些数据库系统中，如果事务运行超过了预设的时间限制，数据库可能会自动提交事务。这通常是为了确保系统不会因为长时间运行的事务而阻塞。
  6. 事务回滚（Rollback）： 如果事务中的操作失败，或者开发者决定不保存事务中的更改，可以通过执行 ROLLBACK 语句来撤销事务。这会撤消事务中所有未提交的更改。

事务的事务传播行为

事务传播行为（Transaction Propagation Behavior）定义了当一个方法被另一个已经存在事务的方法调用时，事务应该如何传播。Spring 框架提供了几种事务传播行为，这些行为决定了新方法执行时是否应该加入当前事务，还是创建一个新的事务，或者完全忽略事务。以下是 Spring 提供的几种事务传播行为：

  1. REQUIRED：这是默认的传播行为。如果当前已经存在事务，则新方法将加入该事务；如果当前没有事务，则新方法将创建一个新的事务。
  2. SUPPORTS：如果当前存在事务，则新方法将加入该事务；如果当前没有事务，则新方法将以非事务的方式执行。
  3. MANDATORY：新方法必须在一个事务中执行。如果当前没有事务，Spring 将抛出 TransactionRequiredException 异常。
  4. REQUIRES_NEW：新方法将创建一个新的事务，并且挂起当前的事务。这意味着即使当前存在事务，新方法也会在一个新的事务中执行。
  5. NOT_SUPPORTED：新方法将以非事务的方式执行，并且挂起当前的事务。如果当前存在事务，它将被挂起，直到新方法执行完成。
  6. NEVER：新方法必须以非事务的方式执行。如果当前存在事务，Spring 将抛出 TransactionRequiredException 异常。
  7. NESTED：如果当前存在事务，则新方法将创建一个嵌套事务；作为嵌套事务的一部分，新方法的执行将受到诸如回合回滚、公共保存点和保存点回滚等嵌套事务框架的支持。
  8. NESTED with SAVEPOINT：这是 NESTED 的一个变体，它允许在嵌套事务中使用保存点（Savepoint）。这使得你可以在嵌套事务中回滚到特定的保存点，而不影响其他部分的事务。

每种传播行为都有其特定的用途，开发者需要根据业务逻辑和性能需求来选择合适的传播行为。例如，REQUIRED 适用于大多数情况，而 REQUIRES_NEW 可能用于需要隔离操作的场景，以避免外部事务的失败影响到新方法的执行。

事务的隔离级别

事务的隔离级别是一个重要的概念，用于控制当多个事务同时对数据库进行操作时，一个事务对数据所做的更改何时对其他事务可见。隔离级别的设置会影响到事务系统的并发性能，不同的隔离级别能够防止不同的数据不一致问题，但也可能引入不同程度的性能开销。

以下是常见的事务隔离级别，由低到高排列：

  1. **读未提交 (Read Uncommitted)**：
    • 事务可以读取未被其他事务提交的更改。
    • 可能导致“脏读”：一个事务读取了另一个未提交事务的更改，如果那个事务回滚，读取的数据将是无效的。
  2. **读已提交 (Read Committed)**：
    • 事务只能读取已被其他事务提交的更改。
    • 解决了脏读的问题，但仍然可能出现“不可重复读”：在同一事务中两次读取同一数据可能会得到不同的结果，因为其他事务可能在两次读取之间修改了数据。
  3. **可重复读 (Repeatable Read)**：
    • 确保在一个事务内多次读取同一数据时，结果是一致的。
    • 解决了不可重复读的问题，但可能会遇到“幻读”：当一个事务在读取某个范围内的记录时，另一个事务插入了新的记录，使得第一个事务在再次读取该范围时看到不一致的结果。
  4. **串行化 (Serializable)**：
    • 最高的隔离级别，事务完全串行执行，避免了脏读、不可重复读和幻读的问题。
    • 这种隔离级别可以通过锁定数据范围来实现，但可能导致较大的性能开销，因为它限制了事务并发执行的能力。

事务回滚规则

指示Spring事务管理器回滚一个事务的推荐方法是在当前事务的上下文内抛出异常。Spring事务管理器会捕捉任何未处理的异常，然后依据规则决定是否回滚抛出异常的事务。默认配置下，Spring只有在抛出的异常为运行时unchecked异常时才回滚该事务，也就是抛出的异常为RuntimeException的子类（Errors也会导致事务回滚），而抛出checked异常则不会导致事务回滚。可以明确的配置在抛出哪些异常时回滚事务，包括checked异常。也可以明确定义哪些异常抛出时不回滚事务。

事务常用配置

● readOnly：该属性用于设置当前事务是否为只读事务，设置为true表示只读，false则表示可读写，默认值为false。例如：@Transactional(readOnly=true)；
● rollbackFor：该属性用于设置需要进行回滚的异常类数组，当方法中抛出指定异常数组中的异常时，则进行事务回滚。例如：指定单一异常类：@Transactional(rollbackFor=RuntimeException.class)指定多个异常类：@Transactional(rollbackFor={RuntimeException.class, Exception.class})；
● rollbackForClassName：该属性用于设置需要进行回滚的异常类名称数组，当方法中抛出指定异常名称数组中的异常时，则进行事务回滚。例如：指定单一异常类名称@Transactional(rollbackForClassName=”RuntimeException”)指定多个异常类名称：@Transactional(rollbackForClassName={“RuntimeException”,”Exception”})。
● noRollbackFor：该属性用于设置不需要进行回滚的异常类数组，当方法中抛出指定异常数组中的异常时，不进行事务回滚。例如：指定单一异常类：@Transactional(noRollbackFor=RuntimeException.class)指定多个异常类：@Transactional(noRollbackFor={RuntimeException.class, Exception.class})。
● noRollbackForClassName：该属性用于设置不需要进行回滚的异常类名称数组，当方法中抛出指定异常名称数组中的异常时，不进行事务回滚。例如：指定单一异常类名称：@Transactional(noRollbackForClassName=”RuntimeException”)指定多个异常类名称：@Transactional(noRollbackForClassName={“RuntimeException”,”Exception”})。
● propagation ：该属性用于设置事务的传播行为。例如：@Transactional(propagation=Propagation.NOT_SUPPORTED,readOnly=true)。
● isolation：该属性用于设置底层数据库的事务隔离级别，事务隔离级别用于处理多事务并发的情况，通常使用数据库的默认隔离级别即可，基本不需要进行设置。
● timeout：该属性用于设置事务的超时秒数，默认值为-1表示永不超时。

事务使用注意事项

  1. 事务注解的正确使用：确保@Transactional注解正确地应用于那些需要事务管理的方法上。如果一个方法调用了另一个事务方法，需要考虑事务的传播行为，以避免不必要的回滚或事务冲突。
  2. 事务的隔离级别：选择合适的事务隔离级别对于避免并发问题至关重要。不同的隔离级别有不同的性能影响，选择不当可能导致性能下降或并发问题。
  3. 事务的传播行为：理解不同的传播行为（如REQUIRED, REQUIRES_NEW, SUPPORTS等）对于事务的控制至关重要。错误的传播行为设置可能导致事务的意外回滚或不应有的事务创建。
  4. 事务的超时设置：设置合理的事务超时时间可以避免事务长时间占用资源。但是，超时时间设置得太短可能会导致事务频繁回滚。
  5. 异常处理：确保正确处理事务中可能抛出的异常。Spring Boot默认会回滚未被捕获的运行时异常，但你需要确保业务逻辑中的错误处理不会意外地导致事务回滚。
  6. 事务的边界：事务的开始和结束应该清晰明确。避免事务跨越多个服务调用，因为这可能导致事务管理变得复杂且难以维护。
  7. 事务的回滚规则：明确哪些异常会导致事务回滚，哪些不会。这可以通过@Transactional注解的rollbackFor和noRollbackFor属性来配置。
  8. 数据库支持：确保你的数据库支持事务。并非所有数据库都支持事务，或者支持的程度不同。
  9. 事务日志：在生产环境中，开启事务日志可以帮助你调试和理解事务行为，尤其是在事务回滚或死锁等问题发生时。
  10. 测试事务：在开发过程中，使用事务测试来确保事务逻辑的正确性。Spring Boot提供了@Transactional注解的testTransaction属性，可以在测试中启用事务。
  11. 事务的并发控制：在高并发环境下，需要考虑事务的并发控制，避免死锁和资源竞争。
  12. 事务的持久化：在分布式系统中，事务的持久化和恢复是一个复杂的问题。Spring Boot提供了对分布式事务的支持，但需要额外的配置和考虑。
  13. 事务的监控：监控事务的执行情况，包括事务的开始、提交、回滚等，对于维护系统的健康状态和性能调优非常重要。
  14. 事务的优化：避免不必要的事务，减少事务的粒度，合理地使用事务的隔离级别，可以提高系统的性能。

事务实践

1.数据库主键冲突

    @Transactional
    public int affairTest1() {
        QueryWrapper<Emp> queryWrapper = new QueryWrapper<>();
        queryWrapper.eq("empno", 7369);
        Emp emp = empMapper.selectOne(queryWrapper);
        System.out.println(emp);  //主键冲突，看是否可以回滚数据
        int insert = empMapper.insert(emp);
        return insert;
    }

注意：这里并没有的回滚，是因为数据库本身设置的主键导致的的报错，所以没有插入成功，不是因为注解。

2.代码bug导致事务回滚

    @Transactional
    public int affairTest2() {
        QueryWrapper<Emp> queryWrapper = new QueryWrapper<>();
        queryWrapper.eq("empno", 7369);
        Emp emp = empMapper.selectOne(queryWrapper);
        System.out.println(emp);
        emp.setEmpno(8000);
        int insert = empMapper.insert(emp);
        int i = 10 / 0; //这里会报错，看数据是否插入成功
        return insert;
    }

注意：可以看到控制台显示插入成功，但是数据并没有插入到表里，发生了事务回滚

3.手动触发事务

@Transactional
    @Override
    public int affairTest3() {
        try {
            QueryWrapper<Emp> queryWrapper = new QueryWrapper<>();
            queryWrapper.eq("empno", 7369);
            Emp emp = empMapper.selectOne(queryWrapper);
            System.out.println(emp);
            emp.setEmpno(8000);
            int insert = empMapper.insert(emp);
            int i = 10 / 0; //这里会报错，看数据是否插入成功
            return insert;
        } catch (ArithmeticException e) {
            System.out.println("发生异常");
            return 0;
        }
    }

注意点：可以看到当我们对异常进行了捕捉后，异常就不会向外抛，所以事务回滚就不生效了。

怎么处理这个问题

 @Transactional
    @Override
    public int affairTest3() {
        try {
            QueryWrapper<Emp> queryWrapper = new QueryWrapper<>();
            queryWrapper.eq("empno", 7369);
            Emp emp = empMapper.selectOne(queryWrapper);
            System.out.println(emp);
            emp.setEmpno(8000);
            int insert = empMapper.insert(emp);
            int i = 10 / 0; //这里会报错，看数据是否插入成功
            return insert;
        } catch (ArithmeticException e) {
            System.out.println("发生异常，手动回滚");
            TransactionAspectSupport.currentTransactionStatus().setRollbackOnly(); 
            return 0;
        }
    }

注意：当我们在catch后添加 TransactionAspectSupport.currentTransactionStatus().setRollbackOnly(); 代码，便可以手动回滚了

4.使用事务时，调用其他子方法进行数据库操作

    public void deal() {
        QueryWrapper<Emp> queryWrapper = new QueryWrapper<>();
        queryWrapper.eq("empno", 7369);
        Emp emp = empMapper.selectOne(queryWrapper);
        emp.setEmpno(9000);
        int insert1 = empMapper.insert(emp);
    }

    @Transactional
    @Override
    public int affairTest4() {
        QueryWrapper<Emp> queryWrapper = new QueryWrapper<>();
        queryWrapper.eq("empno", 7369);
        Emp emp = empMapper.selectOne(queryWrapper);
        System.out.println(emp);
        emp.setEmpno(8000);
        int insert = empMapper.insert(emp);
        temp.deal();
        int a = 1 / 0;
        return insert;
    }

注意：因为 @Transactional 注解应用于整个 affairTest4 方法，所以任何在这个方法内（包括其调用的其他方法）发生的满足回滚条件的异常都将触发整个事务的回滚。在您的情况下，deal 方法是作为 affairTest4 方法的一部分被调用的，因此它的操作也包含在同一个事务中。一旦事务中抛出了指定类型的异常，整个事务，包括 deal 方法中的操作，都会被回滚。

要解决这个问题，您可以：

  1. 分离事务：如果您希望 deal 方法中的操作即使在 affairTest4 方法中发生异常也不回滚，您可以将 deal 方法标记为一个新的独立事务。这可以通过在 deal 方法上添加 @Transactional 注解并设置合适的传播行为（例如 REQUIRES_NEW）来实现。
  2. 调整异常处理：如果 int a = 1 / 0 中的 ArithmeticException 不应该导致事务回滚，您可以在 affairTest4 方法中捕获并处理这个异常，以防止其向上冒泡到事务管理器。

以下时修改后的代码：

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void deal() {
        QueryWrapper<Emp> queryWrapper = new QueryWrapper<>();
        queryWrapper.eq("empno", 7369);
        Emp emp = empMapper.selectOne(queryWrapper);
        emp.setEmpno(9000);
        int insert1 = empMapper.insert(emp);
    }

注意：如果 affairTest4() 和 deal() 方法在同一个类中，并且 affairTest4() 直接调用 deal()，那么 deal() 方法的事务设置可能不会被正确处理。这是因为Spring的事务管理是基于代理的，当从同一个类内部调用另一个方法时，这个代理可能不会被触发。

5.指定出发事务回滚的异常类型

    @Transactional(rollbackFor = IOException.class)
    @Override
    public int affairTest5() throws IOException {
        QueryWrapper<Emp> queryWrapper = new QueryWrapper<>();
        queryWrapper.eq("empno", 7369);
        Emp emp = empMapper.selectOne(queryWrapper);
        System.out.println(emp);
        emp.setEmpno(8000);
        int insert = empMapper.insert(emp);
        someMethod();
        return 0;
    }
        public void someMethod() throws IOException {
        // 抛出IOException
        throw new IOException("文件读取错误");
    }

注意： 添加@Transactional(rollbackFor = IOException.class)后，因为 someMethod()方法抛出IOException，会回滚。

6.自定义事务

@Resource
PlatformTransactionManager platformTransactionManager;
@Resource
TransactionDefinition transactionDefinition;
 

@Override
    public int affairTest6() {
        //开启事务
        TransactionStatus transactionStatus = platformTransactionManager.getTransaction(transactionDefinition);
        try {
            QueryWrapper<Emp> queryWrapper = new QueryWrapper<>();
            queryWrapper.eq("empno", 7369);
            Emp emp = empMapper.selectOne(queryWrapper);
            System.out.println(emp);
            emp.setEmpno(8000);
            int insert = empMapper.insert(emp);
            someMethod();
            //手动提交事务
            platformTransactionManager.commit(transactionStatus);//提交

        }catch (IOException e) {
            //手动回滚事务
            platformTransactionManager.rollback(transactionStatus);//最好是放在catch 里面,防止程序异常而事务一直卡在哪里未提交
            e.printStackTrace();
        }

        return 0;
    }
